---
title: "企业级前端应用框架"
date: "2022-05-20"
---

几年前，前端圈对框架选择这个话题的讨论层出不穷，也出现了很多前端框架，几年后的今天，框架之争已经没那么多，React 和 Vue 成为两大主流，除此之外还有一些大厂自己的框架，比如 Rax 这种，多为 React like 框架。Vue@3.x 支持 Composition API 后摆脱了 this，变得更灵活，React Hooks 在开发者用习惯了之后也变得越来越香，主流的逻辑层前端框架变成了趋于稳定的基础设施，现在越来越多的是应用层的企业级前端框架，比如基于 Vue 的，Nuxt，基于 React 的 Next, Gatsby, 还有国内的 Umi, ice 等等

应用级的框架一般都内置了路由，构建，部署，测试等全方位的解决方案。除了更多开箱即用的API，还有一些框架指定的规范，比如基于文件的路由系统，遵循框架的约定开发也能提高开发效率，减少一些样板代码。Nuxt, Umi, ice 这些框架都支持约定式路由，只是动态路由的标识符有区别，Next, Umi 中使用 `[]`, ice 中使用 `$`, Gatsby 中使用 `{}`，路由的布局组件也是按照约定的组件名来实现，比如 ice 中会检查路面下是否有 `_layout.tsx` 文件，如果有就作为布局组件，该目录下的其他文件作为子组件生成嵌套路由

除此之外应用层框架还提供了一些内置的组件，比如最常见的 Link，Image 组件，通常我们可能会觉得这些应该是由组件库提供的，一个底层框架不需要关心这些，不过这些框架提供 Link，Image 组件只是为了在框架体系下做到更好的性能表现，而不是为了构建漂亮的用户界面，所以这些组件也应该被看做基建的一部分。
